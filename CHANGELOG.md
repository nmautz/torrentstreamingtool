# Changelog

## [3.11.0] — 2026-05-26
- **Major: HTTPS port (443) is now a reverse proxy to the canonical HTTP app, not a second FastAPI instance.** Previously both ports mounted `main:app` and relied on module-global `AppState` being shared across the two `uvicorn.Server` instances inside one process. In practice this produced intermittent state divergence — clients on `https://remote.local` and `http://<lan-ip>` would see different `playing` state until a refresh — because each ASGI mount has its own startup race, SSE buffer, and event-loop scheduling. Replaced with a strict architecture: port 80 serves `main:app` (the only `AppState` in the process), port 443 serves a tiny new `https_proxy:app` that streams every request to `127.0.0.1:80`. There is now provably one source of truth regardless of which hostname or scheme a client uses.
- **New file: `https_proxy.py`.** FastAPI catch-all that forwards method + path + query + headers + streamed request body to the upstream, then streams the response back (so SSE / range requests / large downloads aren't buffered). Strips RFC 7230 §6.1 hop-by-hop headers on both directions. Adds `X-Forwarded-Proto: https`, `X-Forwarded-Host`, and `X-Forwarded-For` so the upstream sees the original client context.
- **`admin_https_redirect` middleware in `main.py` now honors `X-Forwarded-Proto` / `X-Forwarded-Host`.** Without this, every admin hit through the proxy would redirect to `https://127.0.0.1/admin` (loop). With it, `https://remote.local/admin` flows straight through.
- **`run.py` + `daemon._WRAPPER_CONTENT` updated** to point the port-443 uvicorn at `https_proxy:app` instead of `main:app`. The HTTP server still loads `main:app` and runs the full lifespan (qBit login, background tasks). The `lifespan="off"` workaround on the HTTPS server is gone — the proxy app has its own minimal lifespan (creates/closes the upstream `httpx.AsyncClient`).
- **Cost:** an extra ~ms of localhost latency on HTTPS requests for the proxy hop. Negligible for a home server; well worth the elimination of an entire class of sync bugs.

## [3.10.4] — 2026-05-26
- **Fix: Windows service crashed on startup with `RuntimeError: Directory 'static' does not exist`.** Task Scheduler launches the wrapper with CWD = `C:\Windows\System32`, but `main.py` mounts `StaticFiles(directory="static")` with a *relative* path. Result: every `uvicorn.Server` raised on import, the supervisor restarted 5× in 1.0s, hit the fast-death circuit breaker, and the service gave up. Fixed by `os.chdir(HERE)` at the top of `streamlink_service.py` (and the matching `_WRAPPER_CONTENT` template in `daemon.py`) so all relative paths in `main.py` resolve next to the wrapper. macOS launchd and Linux systemd were unaffected because their unit files already set `WorkingDirectory={HERE}` — the bug only surfaced on Windows where `schtasks` has no equivalent.

## [3.10.3] — 2026-05-26
- **Fix: playing media state not shared between HTTP and HTTPS connections; service shuts down immediately on Windows.** Both `run.py` and `streamlink_service.py` were spawning two separate `uvicorn` subprocesses (port 80 HTTP, port 443 HTTPS) with independent `AppState`. Fixed by running both servers concurrently in the same `asyncio` event loop so they share state. Also fixed two service-specific failures: (1) uvicorn's default `dictConfig` adds `StreamHandler`s pointing at `sys.stderr`/`stdout`, which are `None` in a Task Scheduler service (no console) — suppressed via `log_config={"version":1,...}`; (2) the supervisor loop broke on `rc=0` treating a fast startup crash as a clean exit — fixed so any `asyncio.run()` return (non-interrupt) triggers a restart.

## [3.10.2] — 2026-05-26
- **Fix: auto-update was blocked on Windows by a UAC prompt during the "reinstall service" step.** `daemon.install()` on Windows requires admin and tries to UAC-elevate via `ShellExecute(..."runas"...)`; from a service-launched uvicorn there's no interactive desktop, so the elevation either fails or sits waiting for a click that never comes — breaking the "fully automatic" requirement. **Root cause:** we were doing more than necessary. The OS service entry (Task Scheduler task / launchd plist / systemd unit) points at `streamlink_service.py` by *path*, and that path is stable across versions. The thing that actually needs to change between versions is the *contents* of that wrapper script. Writing a file inside the project dir needs no elevation.
- **New flow:** Step 3 of `_run_apply` was changed from `daemon.uninstall() + daemon.install()` to `updater.refresh_service_wrapper()`, which just regenerates `streamlink_service.py` from the freshly-pulled `daemon._WRAPPER_CONTENT`. Plain file write, no admin, no UAC. The OS supervisor keeps finding the same wrapper path; when the reboot brings everything back up it runs the new wrapper code on its own. Idempotent: if the wrapper file is already byte-identical, the rewrite is skipped and the diagnostic panel logs "Wrapper already up to date" instead.
- **Backend:** `updater.reinstall_service()` removed; replaced with `updater.refresh_service_wrapper()` (no `daemon.uninstall()` / `daemon.install()` calls). Phase id `reinstalling-service` renamed to `refreshing-service`; the user-side banner phase set + the admin UI phase message updated. The response field `service_reinstalled` kept for compatibility with the existing toast branch — true means the wrapper write succeeded.
- **When the admin still has to run `python run.py --install` manually:** only when `daemon.py` itself changes its OS-service registration logic (a new plist key, different `schtasks` args, etc.). The wrapper regeneration covers content-only changes, which is the vast majority of updates. If a future release needs a re-registration we'll call that out in the release notes.
- **Docs:** `docs/ADMIN.md` Step 3 description, `docs/API.md` apply-endpoint shape, `docs/GOTCHAS.md` Windows-elevation entry all updated to describe wrapper refresh instead of reinstall.

## [3.10.1] — 2026-05-26
- **Fix: `setup.py` exited rc=1 from the auto-updater on Windows.** Root cause: Python 3.13 on Windows defaults `sys.stdout` to the host's legacy ANSI code page (cp1252 in en-US) when stdout is a pipe, and setup.py's first banner contains the Unicode box-drawing glyphs `┌─┐` — those are unencodable in cp1252, so the very first `print()` raised `UnicodeEncodeError` and the whole process bailed before doing anything. The actual update worked (git pull succeeded) but `_run_apply` correctly refused to proceed past a failed setup.
  - **setup.py** now reconfigures `sys.stdout` + `sys.stderr` to UTF-8 with `errors="replace"` at startup (idempotent; no-op when already UTF-8 — i.e. on a real terminal or POSIX). Any future glyphs degrade to `?` instead of crashing the script.
  - **updater.run_setup** also passes `PYTHONIOENCODING=utf-8` in the subprocess env as belt-and-braces — fixes the same crash even on a setup.py that predates the in-script reconfigure.
- **Make non-interactive re-runs of `setup.py` robust to transient pip / network failures.** New `STREAMLINK_AUTOUPDATE=1` env var (set by `updater.run_setup`) tells setup.py:
  - `pip install` failures are logged as warnings and continue — they don't kill the update. A transient PyPI hiccup shouldn't burn an apply; the existing venv contents are good enough to come back up, and the next apply will catch up.
  - Skip OS-app installs (`install_core_deps`, `install_jackett_service`, `install_smart_skip_deps`) — winget/brew need an interactive desktop and can hang or fail from a service account. Those run only during the initial interactive setup.
  - Skip `offer_service_install()` — the updater calls `daemon.uninstall()` + `daemon.install()` itself right after setup, so re-running it here would risk a Windows UAC prompt blowing up the run (no interactive desktop).
  - Force `reuse_env=True` when an existing `.env` is found — never prompt for re-entry of values during an automated update.
- **Better error surfacing in the admin UI.** When setup.py fails, the apply phase message now appends the last non-empty line of setup.py's output (typically the exception class + message) so the admin sees the actual cause without having to expand the collapsed diagnostic panel. Previously you'd see `setup.py exited rc=1` and have to click to dig.

## [3.10.0] — 2026-05-26
- **Smart Skip now records *why* fingerprinting failed and surfaces it to everyone.** Previously, if `analyze_series` couldn't extract a fingerprint for a file (corrupted audio, unsupported codec, missing ffmpeg/fpcalc, file moved off disk, no peers, etc.) the file was silently dropped from the result and users got no Skip Intro/Skip Credits offer with zero indication why. Now every file the analyzer touches gets a `skip_data` entry — successes carry the usual intro/credits, failures carry `analysis.source = "failed"` with an `error_code` and human-readable `error`. The user-facing library list shows a "⚠ Intro/credits skip not available" chip on items where every file failed (and a softer "Skip partial" chip when only some failed). The admin Smart Skip tab's editor shows a red error block with the full message above each failed file's time inputs, and a new **Fingerprint Log** panel under the item list streams the most recent 200 events (timestamp + error code + filename + message) — refreshes automatically when any in-flight analysis hits a terminal state.
- **Failed files retry on their own.** `_schedule_series_analysis_if_eligible` now treats `analysis.source == "failed"` as eligible for re-analysis, so when a new sibling episode in the series flips to ready, the larger fingerprint pool gets another shot at the previously-failed file — no admin click needed. Manual overrides (`source == "manual"`) are still never overwritten.
- **Backend:** new `analyzer.ERR_*` constants for the seven failure modes (`no_binary`, `file_missing`, `no_duration`, `fp_empty`, `too_short`, `no_skip_points`, `exception`); `analyze_series` always returns one entry per input file. `state.analyzer_log` is a 200-deep `deque` ring buffer fed by a new `_log_analyzer_event` helper. `_run_series_analysis` persists every entry (including failures) into `library.json` then logs one event per failed file. New `GET /api/admin/analyzer-log?limit=N` returns the buffer; `/api/library` adds `skip_status ∈ {ok, partial, failed, pending, none}`; `/api/admin/library` adds `files_failed` and `skip_status`; `/api/admin/library/{id}/skip-data` adds `error_code` and `error` per file. The series-level `analysis_jobs[key].status` now distinguishes "all failed" from "partial success" in its message.
- **Docs:** [docs/ANALYZER.md](docs/ANALYZER.md) gets a new "Failure tracking" section with the full error-code table and downstream consumer list; [docs/LIBRARY_DATA.md](docs/LIBRARY_DATA.md) updates the `skip_data` schema; [docs/API.md](docs/API.md) lists the new endpoint and the augmented field set on the existing ones.

## [3.9.1] — 2026-05-26
- **Apply Now is now full-cleanup: git pull → setup.py → uninstall + reinstall the OS service → reboot the host.** Replaces the previous "SIGTERM uvicorn, hope the supervisor restarts" sequence — the service wrapper (`streamlink_service.py`) is regenerated from the freshly-pulled `daemon.py` before the reboot, and a full host reboot gives the new code a clean process tree on the way back up. Live phase IDs are now `applying → setup → reinstalling-service → rebooting`; the user-side banner tints red while the host is in `rebooting` so non-admin viewers see the disruption coming.
- **Switch back to an older branch from Apply Now in one click.** Apply Now now sends the branch picker's *current* value (not the saved config), so `alpha → main` is one confirm-gated click; the confirmation explicitly names the direction ("Switching working tree 'alpha' → 'main'"). `git switch -C origin/<branch>` + `git reset --hard` is symmetric in both directions. `library.json`, `.env`, `.offline_cache/`, `.background/` are all gitignored and survive the switch.
- **Backend:** new `updater.reinstall_service()` runs `daemon.uninstall()` + `daemon.install()` on a worker thread, captures their stdout into the diagnostic buffer, and returns `{ok, output, error}`. `_run_apply` rewired to: git → setup → reinstall (best-effort: a failed reinstall is logged but the reboot still fires — better to come back up with a stale service than be stuck pre-reboot) → `_reboot_machine(delay=1.5)`. `_kill_self_for_restart` removed (reboot supersedes it). Request model field renamed `restart` → `reboot`.
- **Docs:** GOTCHAS notes that the auto-apply path *requires* host auto-login (the reboot ends at a login screen otherwise — the user-level service never starts). ADMIN.md + API.md updated for the new apply shape (`reboot` flag, `service_reinstalled` field, new phases).

## [3.8.1] — 2026-05-26
- **Fix: the global stream-prep pill leaked titles of admin-only items to profiles that couldn't otherwise see them.** Anyone polling `/api/offline-active` got the real item title in the pill detail line (e.g. *"Avatar S2E5 · 4m 12s left"*), even when their profile was blocked from the item by the Content Lock. ## [3.8.0] — 2026-05-25
- **New setting: "YouTube Starting Volume".** Configures the host OS volume that's pre-set **before** the YouTube kiosk loads, so the first audio frame plays at a known level instead of whatever the OS happened to be at (often max). Lives in the profile-settings panel next to "System Volume After YouTube" (default **30 %**). Stored at `library.json → settings.youtube_start_volume`.
- **Order matters and is enforced.** `/api/youtube` now: (1) snapshots the current OS volume for the Stop restore, (2) **sets the OS mixer to the configured start volume**, (3) takes over now-playing state, (4) `pl_stop`s VLC, (5) broadcasts `yt_command:load`, (6) launches the kiosk. The pre-set happens before the load broadcast and before Chrome paints, so the IFrame player can never out-run it. `state.vlc_volume` is seeded from the pre-set so the dashboard slider lines up with reality the moment YouTube takes over.
- **Backend:** new `_youtube_start_volume`, `YouTubeStartVolumeReq`, `GET`/`POST /api/settings/youtube-start-volume`. **Frontend:** new red slider in the profile-settings panel with matching load/save plumbing (`saveYoutubeStartVolume`).

## [3.7.4] — 2026-05-25
- **Fix: "COM method call without VTable" spam in the logs after each volume change** (`Exception ignored in <function _compointer_base.__del__>`). Order-of-teardown bug: the `finally` block in `_windows_volume_op` calls `CoUninitialize` *before* Python destroys the function's frame, so the local COM pointers' `__del__` calls `Release()` against a torn-down apartment. Volume changes were still landing — just every successful call left three ignored exceptions in the logs. Fix: do all COM work inside an inner closure (`_do_com_work`); when it returns, its frame is destroyed first, the COM pointers `Release()` while the apartment is still alive, *then* the outer `finally` runs `CoUninitialize`. `op(vol)` returns a plain `bool`/`int` so no COM refs leak out across the boundary.

## [3.7.3] — 2026-05-25
- **Fix: `'AudioDevice' object has no attribute 'Activate'`** — even with `comtypes` installed, Windows volume control still failed because **`pycaw.AudioUtilities.GetSpeakers()` is API-unstable across pycaw versions**. Some releases return the raw `IMMDevice` COM pointer (has `.Activate()`), others return a Python `AudioDevice` wrapper that doesn't. `_windows_volume_op` now **bypasses `AudioUtilities.GetSpeakers()` entirely**, going through `CoCreateInstance(CLSID_MMDeviceEnumerator) → GetDefaultAudioEndpoint(eRender, eMultimedia) → Activate(IAudioEndpointVolume)` directly. This reuses pycaw's COM **interface definitions** (`pycaw.api.endpointvolume.IAudioEndpointVolume`, `pycaw.api.mmdeviceapi.IMMDeviceEnumerator`) — which have been stable — but not its convenience wrapper. Works against any pycaw release we ship.
- Also falls back to a hard-coded `CLSID_MMDeviceEnumerator` GUID (`BCDE0395-E52F-467C-8E3D-C4579291692E`) + numeric `EDataFlow`/`ERole` constants if pycaw ever moves them, so the path keeps working through future renames.

## [3.7.2] — 2026-05-25
- **Fix: Windows volume control still failed — `No module named 'comtypes'`.** `pycaw` declares `comtypes` as a transitive dep, but on real hosts (the user's box, in particular) pycaw can end up installed without comtypes — older standalone installs, `pip install --no-deps`, partial venvs, etc. Now `comtypes>=1.2.0` is pinned **explicitly** in `requirements.txt` under the same `sys_platform == "win32"` marker (belt-and-braces), so a clean `pip install -r requirements.txt` always lands both.
- **Better diagnostic when an import is missing.** The error message now names the *actual* missing module (via `ImportError.name`) — so the dashboard toast says *"Windows volume control dep `comtypes` is not installed"* instead of the generic *"pycaw is not installed"*, telling the operator exactly which `pip install` to run.

## [3.7.1] — 2026-05-25
- **Fix: YouTube volume controls did nothing on Windows.** Two failure modes, both silent:
  - **COM was not initialized on the thread that called pycaw.** Volume operations run via `asyncio.to_thread`, which dispatches to Python's default `ThreadPoolExecutor` — those worker threads don't have COM initialized, so `AudioUtilities.GetSpeakers()` raises *"CoInitialize has not been called"*, the helper's `except` swallowed it as a warning, and the dashboard slider just no-oped. Volume helpers now call `comtypes.CoInitialize()` / `CoUninitialize()` around each operation (`_windows_volume_op` wrapper), so they work regardless of which pool worker happens to run them.
  - **`pycaw` may not be installed at all** if the user upgraded without re-running `pip install -r requirements.txt`. The `ImportError` got the same generic "volume failed" treatment. Now the import failure is detected explicitly and surfaces a clear message — *"pycaw is not installed on the server. Run `pip install -r requirements.txt` (or re-run setup.py) and restart."* — both in `logs/streamlink_app.log` and as a one-shot toast in the dashboard.
- **Failures now surface to the dashboard instead of silently no-opping.** `POST /api/youtube/control` returns **503** with `{ok:false, error}` when the volume call failed, and `ytControl` (JS) shows the server's error message via a one-shot session-scoped alert so the user isn't left guessing why the slider doesn't work.

## [3.7.0] — 2026-05-25
- **New: Clear All button on the admin Server Logs card.** Next to **Refresh** / **Download All (.zip)**, a confirm-gated **Clear All** wipes every file in `logs/` — handy after pulling diagnostics off a remote box, so the next bundle isn't 100 MB of stale history.
- **Backend:** `DELETE /api/admin/logs`. The live rotating handlers (`streamlink_app.log`, `hls.log`) are **truncated in-place** via `handler.acquire()` + `handler.stream.truncate(0)`, not deleted. Deleting was the obvious approach but wrong on both platforms we care about: on **Windows** you can't `unlink` a file the running process has open for writing, and on **POSIX** a delete leaves logging's FD valid but disconnected from any on-disk file — subsequent writes silently vanish until the next restart. Rotated siblings (`.1`/`.2`/`.3`) and the `streamlink.err` written by launchd/systemd/the Windows service are unlinked; if `unlink` fails (service still holds an exclusive Windows handle), we fall back to a write-mode truncate so the user still sees the file empty. Returns `{ok, cleared:[name], errors:[{file,error}]}`.

## [3.6.1] — 2026-05-25
- **Fix: YouTube kiosk launched on Windows but `/tv` never checked in.** The logs showed Edge launching every play attempt followed by the 12 s heartbeat watchdog firing — the page wasn't loading. Cause: the kiosk URL was `http://localhost/tv?v=…`, and Windows' hosts file resolves localhost to **both `::1` and `127.0.0.1`** with IPv6 first. uvicorn binds `0.0.0.0:80` (IPv4 only), so Chromium hit ECONNREFUSED on `::1` and either showed an error page or stalled past the watchdog window. The kiosk URL now pins **`http://127.0.0.1/tv?v=…`**, sidestepping the dual-stack mess entirely. Added Edge first-run/signin/welcome-modal suppression flags (`--disable-fre`, `--disable-features=msImplicitSignin,SigninInterceptBubbleV2,…`, `--noerrdialogs`) so a fresh `--user-data-dir` profile can't get blocked on a welcome screen either.
- **Fix: YouTube played at system max volume; dashboard slider was a no-op.** The IFrame player's `setVolume` only scales the audio *before* the OS mixer — the actual loudness is whatever the host's system volume is set to. The dashboard volume slider now drives the **OS system volume** during YouTube (via `pycaw` on Windows, `osascript` on macOS, `pactl`/`amixer` on Linux), and the IFrame is locked at 100 % unmuted so the OS mixer is the sole amp. While YouTube is on, the slider's range flips to **0–100** (OS scale) instead of VLC's 0–200.
- **New setting: "System Volume After YouTube".** The host's OS volume is **restored to a configured target** when YouTube stops — so headphones at 100 % don't blow eardrums and a movie session doesn't leave the room loud the next time anyone touches VLC. Lives next to "Max Volume" in the profile-settings panel (default 70). The restore only runs after the kiosk process is confirmed gone (polled with a 4 s deadline) so we never twist the volume knob underneath the still-playing video.
- **Backend:** `_set_system_volume_sync` / `_get_system_volume_sync` (pycaw on Windows, osascript on macOS, pactl/amixer on Linux), `_system_volume_default`, `state.system_volume_before_yt` snapshot, `POST /api/youtube/control` now handles `volume_set` / `volume_step` server-side (no longer broadcast as `yt_command`), `/api/stop` restores the OS volume after kiosk teardown, new `GET`/`POST /api/settings/system-volume-default`, new `library.json → settings.system_volume_default` (0-100, default 70), `pycaw` dep added to `requirements.txt` under a `sys_platform == "win32"` marker.

## [3.5.2] — 2026-05-25
- **Fix: on Windows the YouTube kiosk launched but stayed behind VLC** — the idle background video kept the screen and the kiosk window only flashed in the taskbar until you clicked it. Two causes, both Windows-specific:
  - **Focus-stealing prevention.** The server isn't the foreground process, so the browser window it spawns is denied focus (Windows flashes its taskbar button instead). Now `_bring_tv_to_front` finds the kiosk window **by its title** (`_TV_WINDOW_MARKER` = the `static/tv.html` `<title>`, since Chrome is multi-process so PID matching is unreliable) and forces it forward with the same cocktail used for VLC (zero foreground-lock timeout + synthetic ALT + `AttachThreadInput` + `SetForegroundWindow` + clear taskbar flash), retrying for ~10 s while the window is still being created.
  - **VLC stayed on top.** `/api/youtube` only `pl_stop`ped VLC; its window remained. Now it also **minimizes VLC** (`vlc_minimize`, via `_bring_tv_to_front`). Safe because the return-to-background path (`_play_background_video → vlc_focus_and_fullscreen`) `SW_RESTORE`s it.
  - **The background focus loop no longer fights the kiosk.** `vlc_focus_and_fullscreen` now bails the moment `youtube_active` is set — previously a still-running loop would keep minimizing the kiosk and re-focusing VLC for up to ~24 s after a background session.

## [3.5.1] — 2026-05-25

  - **Reboot Machine.** New `POST /api/admin/reboot` restarts the whole host (not just the web server). `_reboot_machine()` tries platform-appropriate commands in order — macOS: System Events restart (no sudo from a launchd user agent) → `sudo -n shutdown -r now` → `shutdown -r now`; Linux: `systemctl reboot` → `sudo -n shutdown` → `shutdown`; Windows: `shutdown /r /t 0`. Fires ~0.5 s after the response flushes so the UI gets confirmation first.
  - **Scheduled Restart.** New `scheduled_reboot_loop` background task + `GET`/`POST /api/admin/scheduled-reboot`. Config (`enabled`, `time` HH:MM, `timezone`, `idle_minutes`) persists under `library.json → settings.scheduled_reboot`. At the configured local time (default 00:00 America/Los_Angeles) it checks whether the box was used in the last `idle_minutes`; if idle it reboots, otherwise it waits `idle_minutes` and re-checks, repeating until idle. A persisted `last_fired` date guards against the just-rebooted machine immediately re-arming and looping.
  - **Idle detection.** New `track_activity` middleware stamps `state.last_activity` on mutating requests + search (routine GET polling is ignored). `_machine_in_use()` also treats live VLC playback/pause of non-background content and any active stream/download as "in use", so a reboot never interrupts active watching or a running download.
  - **Auto-recovery requirement.** For the host to come back after a reboot, the user must configure OS auto-login and install the StreamLink system service (`run.py --install`). Documented in the README and the System tab.

## [2.9.0] — 2026-05-20
- **Jackett reliability: detect and recover a hung indexer instead of needing a reboot.** Root cause of "Jackett stops working after a while, only a reboot fixes it": the watchdog's liveness check was a bare TCP port-open, and a hung Jackett keeps its listener socket bound while it stops serving HTTP — so the watchdog thought it was healthy and never restarted it. Even when it did try, `sc.exe start Jackett` is a no-op (1056 ALREADY_RUNNING) on a wedged service, so only a reboot cleared it.
  - **HTTP health check.** The watchdog, `run.py` startup, and a new in-app monitor now probe `GET {INDEXER_URL}/UI/Login` (any HTTP status proves the web stack is alive) via `_http_ok()` instead of a port check. `ServiceSpec` grew a `health_check` hook; `is_alive()` uses it for Jackett (VLC/qBit unchanged), and `start()` waits on it so a restart doesn't tight-loop before Jackett finishes warming up.
  - **Force-down before restart.** A new `pre_restart` hook force-stops the wedged Jackett (`sc stop` + wait for STOPPED, hard-kill fallback; `kill_by_name` off-Windows) before relaunching, so the port frees and the restart actually takes. When no `Jackett` service exists, the watchdog now launches the tray/console exe as a user process instead of giving up.
  - **Permission grant so a non-elevated watchdog can recover the service.** A LocalSystem `Jackett` service can only be `sc` stopped/started by an admin — the reason a non-elevated StreamLink hit a UAC/access-denied wall. `setup.py`'s new `grant_jackett_service_control()` additively grants Authenticated Users `SERVICE_START`+`SERVICE_STOP` via `sc sdset` and sets `sc failure` auto-restart actions (idempotent, best-effort, one UAC prompt; prints the manual commands if elevation is declined). Re-run `setup.py` to apply. Access-denied paths now log an actionable hint.
  - **In-app catch-safe + visibility.** New `jackett_health_monitor` background task polls every 20 s, publishes `state.jackett_ok` (in `state_snapshot()`) and a `jackett_status` SSE event, and — only after ~2 min of sustained local outage (i.e. when no watchdog has recovered it) — calls the reusable `watchdog.restart_jackett()` as a backstop, so it never duels a healthy watchdog.

## [2.8.0] — 2026-05-19
- **Minor feature: bidirectional Handoff (TV ⇄ device), time-synced.**
  - **TV → device.** While a library item is playing on the TV (VLC), a new emerald **Device** button (next to Stop in the player footer, and a **To Device** tile next to Stop in the fullscreen controls) hands playback to the browser that tapped it. Both require a **0.5 s hold** to activate (same `_holdStart` / `.hold-btn` progress-fill pattern as Stop), so an accidental tap can't yank playback off the TV. The button shows only during library playback (`is_library_playback && library_item_id`) and is gated by `withInflight("handoff")` against double-taps. Readiness is resolved from `prepFileState` (instant when known) and a per-file `GET /api/library/{id}/prep-status` check (`_handoffReadyState` / `_maybeRefreshHandoffReady`), and the button flips to active automatically once the file is prepped (via the episode picker's Prep / a card's Prep for Streaming). Tapping it while not prepped shows a toast instead of acting.
  - **Device → TV.** The on-device local player now has the mirror-image **To TV** button in its fullscreen header (next to Stop): `lpHandoffToVlc` captures the `<video>` position + remaining playlist tail, tears down the device player (flushing progress), and starts VLC playback at the exact same time via `playLibraryFiles` (`POST /api/library/{id}/play` with `seek_first_to`). Gated by `withInflight("handoff_vlc")`.

## [2.7.0] — 2026-05-19
- **Minor feature: the auto-skip countdown now runs *before* the skip point, and the intro/credits is skipped in full.** Previously the countdown started *at* the intro/credits start and skipped N seconds later, so the first 5 s of an intro (or 10 s of credits) played first. Now the countdown leads *into* the skip point — e.g. for an intro starting at 1:30 the 5-second countdown runs 1:25→1:30 and the skip fires exactly at 1:30, so none of the intro plays. Credits likewise count down over the 10 s before `credits_start` and advance to the next episode the moment credits begin. The countdown is now **position-driven** (`_run_skip_countdown` shows `ceil(target − pos)` and fires at `pos ≥ target`), so it tracks real playback: it freezes while paused and follows seeks, polling every 0.5 s so the popup clears within ~0.5 s when you scrub out of the window. The auto trigger window widened to `[skip_point − lead, …)`; the manual (auto-skip-off) Skip button is unchanged.

## [2.6.2] — 2026-05-19
- **Bug fix:** The countdown popup ("Skipping intro in 1") stayed frozen on the TV after the skip fired — and after any seek that ended the countdown. Root cause: VLC's `marq` filter reads `--marq-file` with `getdelim()`, which returns EOF on an *empty* file, so the filter **keeps the previously-rendered text** instead of blanking it (and logs a read error every refresh tick). Every "clear" wrote an empty string, which was a no-op on screen. `_marquee_write` now clears by writing a single space — a valid non-empty line that forces marq to update but renders no glyph (we draw no background box). `run.py` / `watchdog.py` likewise seed the file with a space, not "". This is what actually made 2.6.1's two-sided-window abort visible on screen.

## [2.6.1] — 2026-05-19
- **Bug fix:** The "Skipping intro in N" countdown popup no longer lingers when the viewer scrubs the progress bar past the intro. `_run_skip_countdown` previously only aborted when the position moved *below* the window floor (a backward seek); a forward seek out of the intro left the popup up and would have yanked playback back to the intro end when the count hit zero. The countdown now takes a `[floor, ceil)` window (intro `ceil = intro end`; credits unbounded) and aborts on leaving it in **either** direction, clearing the popup within ~1 s of the seek.

## [2.2.1] — 2026-05-18
- **Bug fix:** Player UI no longer gets stuck on "Loading…" / buffering after a library play, even though VLC is already playing. The 2.2.0 refactor of `/api/library/{id}/play` flipped `state.stream_status` to `"playing"` only *after* the full `in_play` + sequential `in_enqueue` loop completed — so a "continue watching" play that enqueued 50+ remaining episodes left the UI in buffering for many seconds even though VLC had been playing the first episode from the start. ## [2.2.0] — 2026-05-18
- **Minor feature: slow-network playback responsiveness.** On a slow or flaky link the app used to feel dead for several seconds between "user clicks Play" and "VLC actually opens the file" — the Play response was synchronous on the VLC HTTP roundtrips, the chooser modal closed immediately, and the player UI didn't paint until the SSE `playing` event arrived. Four-part fix:
  1. **Server: non-blocking playback handoff.** `/api/library/{id}/play`, `/api/vlc/prev`, `/api/vlc/next`, `/api/stop`, and `/api/stream` now broadcast a state event (`buffering` / `idle`) and return 202 immediately. The slow VLC `in_play` + `in_enqueue` work (and the qBit deletes on stop/stream) run in background tasks so the response no longer waits on the network at all. A new `state.library_play_task` is tracked and cancelled on subsequent Play / Stop so a slow handoff can't race a newer action.
  2. **Client: optimistic buffering UI.** A new `_optimisticBuffering(label, itemId)` helper flips `app.stream_status` to `buffering`, populates `active_title` from the cached library item, and opens the mobile fullscreen overlay the instant the user clicks Play — no waiting for the server. `renderPlayer` now also auto-opens the overlay on `stream_status === "buffering"` (previously only on `playing`). For library plays the buffering badge says **"Loading…"** and the fullscreen status says **"Starting playback…"** (instead of the misleading "Buffering…" / "Connecting…" copy, which made fully-downloaded episodes look like network problems).
  3. **Client: in-flight guards on Play.** `continueLibraryItem` and `playLibraryFiles` now run under `withInflight("play_${itemId}", …)`, so an impatient double-tap while the server is mid-handoff is dropped client-side instead of racing extra `in_play` requests to VLC. They also bail with a warn-toast if `app._connected === false` (i.e. SSE has been disconnected past the 4 s grace), since the response would otherwise never get a confirming state event.
  4. **Client: visible SSE connection state.** The navbar's SSE pill (`#sseLabel`) now shows **LIVE** / **OFFLINE** on mobile too (previously `hidden sm:block`), and after a 4 s reconnect grace the user gets a "Lost connection to host — reconnecting…" toast plus a guard that blocks new Play actions until SSE re-opens.

## [2.1.4] — 2026-05-16
- **Bug fix (Windows):** Dropped `/RL HIGHEST` from the scheduled task created by `run.py --install`. On Windows, ports below 1024 do not require admin to bind (the "privileged ports" rule is Unix-only), so the wrapper doesn't need elevation. HIGHEST was actively harmful for Standard User accounts — it made Task Scheduler try to elevate their token at trigger time, which failed silently and left the task registered but never running. Firewall rules (which DO need admin) are now added during `_windows_install` while we hold the UAC-granted admin token.
- **Setup warning:** `setup.py` now detects per-user Python installs on Windows (Microsoft Store python, `AppData\Local\Programs\Python`, etc.) and warns that the resulting venv will not be usable by any other Windows user account. This was the silent root cause of the service-doesn't-run case: when the scheduled task ran as a different user, the venv launcher couldn't read the base python in the installer's user profile and bailed with "Access is denied" before the wrapper could log anything.
- **Docs:** New GOTCHAS entries for Microsoft Store Python and the `/RL HIGHEST` / `/RU` Windows-task pitfalls so the next person doesn't have to re-debug.
