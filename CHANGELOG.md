# Changelog

## [3.6.0] — 2026-05-25
- **New: download the server's log files from the admin panel.** A **Server Logs** card now sits in the System tab and lists every file under `logs/` (`streamlink_app.log`, `hls.log`, `streamlink.err`, plus the rotated `.1`/`.2`/`.3` siblings) with size + last-modified, newest first. Each row has a per-file **Download** link, plus **Download All (.zip)** at the top that streams a deflate-compressed bundle named `streamlink-logs-YYYYMMDD-HHMMSS.zip`. Lets operators pull diagnostics off a remote box (Windows in particular) without needing SSH or Remote Desktop.
- **Backend:** `GET /api/admin/logs` (listing), `GET /api/admin/logs/{name}` (single file, attachment disposition), `GET /api/admin/logs/_bundle` (streaming ZIP via the same `os.pipe()` pattern used by `/api/library/{id}/zip`). All three require admin auth, and `_safe_log_path` rejects any name containing a slash, `..`, an absolute path, or a resolved location outside `LOG_DIR` so the file-by-name endpoint can't be coaxed into reading anything outside `logs/`. The per-file route uses the token via `?admin_token=` query param so plain `<a download>` anchors work (anchor downloads can't set headers).

## [3.5.2] — 2026-05-25
- **Fix: on Windows the YouTube kiosk launched but stayed behind VLC** — the idle background video kept the screen and the kiosk window only flashed in the taskbar until you clicked it. Two causes, both Windows-specific:
  - **Focus-stealing prevention.** The server isn't the foreground process, so the browser window it spawns is denied focus (Windows flashes its taskbar button instead). Now `_bring_tv_to_front` finds the kiosk window **by its title** (`_TV_WINDOW_MARKER` = the `static/tv.html` `<title>`, since Chrome is multi-process so PID matching is unreliable) and forces it forward with the same cocktail used for VLC (zero foreground-lock timeout + synthetic ALT + `AttachThreadInput` + `SetForegroundWindow` + clear taskbar flash), retrying for ~10 s while the window is still being created.
  - **VLC stayed on top.** `/api/youtube` only `pl_stop`ped VLC; its window remained. Now it also **minimizes VLC** (`vlc_minimize`, via `_bring_tv_to_front`). Safe because the return-to-background path (`_play_background_video → vlc_focus_and_fullscreen`) `SW_RESTORE`s it.
  - **The background focus loop no longer fights the kiosk.** `vlc_focus_and_fullscreen` now bails the moment `youtube_active` is set — previously a still-running loop would keep minimizing the kiosk and re-focusing VLC for up to ~24 s after a background session.

## [3.5.1] — 2026-05-25
- **Fix: YouTube-on-TV never started on Windows** (it worked on macOS). The video failed to launch and you'd see the idle background video stop, VLC flash its loading animation, then the background video restart — the tell-tale sign that the kiosk launch failed: `_launch_tv_browser` returned False, so `/api/youtube` hit its 500 path, reset `youtube_active`, and ~3 s later `background_video_loop` reclaimed VLC.
  - **Root cause: browser discovery was too narrow on Windows.** It only checked three hard-coded Program Files paths and silently returned None when Chrome/Edge lived elsewhere — most commonly a **per-user Chrome install under `%LOCALAPPDATA%`**, or Edge at a path the list didn't cover.
  - **Fix:** `_find_chrome` now casts a wide net on Windows — the **`App Paths` registry** (HKCU + HKLM, for `chrome.exe` / `msedge.exe` / `brave.exe` / `chromium.exe`), per-user `%LOCALAPPDATA%`, both `Program Files` trees, and PATH — covering Chrome, **Edge** (preinstalled on Win10/11, so this essentially always resolves), Brave and Chromium. Discovery and launch now **log** what they found / why they failed (`logs/streamlink_app.log`) instead of failing silently.
  - **Plus a health-check:** after launching the kiosk, if the `/tv` page doesn't heartbeat within 12 s (it normally checks in within ~1 s), the server now reports a clear on-screen error — "YouTube didn't start on the TV — the browser failed to open" — instead of silently dropping back to the background video. Catches locked-profile / instant-exit / session-0-service cases too.
  - **CLAUDE.md:** added a top-of-file directive that **Windows is the primary target platform** (then Linux, then macOS) — a feature that works on macOS but not Windows is a bug, not partial success.

## [3.5.0] — 2026-05-25
- **New: play any YouTube link on the TV, controlled from the dashboard.** A red **Play on TV** input now sits at the top of the Search tab — paste a YouTube link (watch, `youtu.be`, Shorts, live, embed, or a bare id) and it opens fullscreen on the TV (the host display) and starts playing.
  - **Plays in a browser, not VLC.** VLC 3.0's bundled `youtube.lua` is broken (it "plays" the watch page for a few seconds, never resolves a title/length, then stops), and yt-dlp-into-VLC adds a fragile dependency and caps at 720p without an audio-slave hack. Instead the video plays in a fullscreen **Chrome kiosk** on the host via the **YouTube IFrame Player API** (`static/tv.html`), so quality/ads/captions all "just work".
  - **Driven by the existing controls.** The footer + fullscreen play/pause, ±10/30 s, scrubber, and volume all route to the TV's YouTube player when it's the active playback (`app.youtube_active` → `POST /api/youtube/control` → SSE `yt_command` → IFrame API). The `/tv` page reports position/duration/title/volume back every second (`POST /api/youtube/tv-state`), mirrored onto the reused display fields so the footer scrubber renders YouTube exactly like a VLC play. Save-to-library, handoff, episode-nav and track menus are hidden in YouTube mode.
  - **Backend:** `POST /api/youtube`, `POST /api/youtube/control`, `POST /api/youtube/tv-state`, `GET /tv`; `_extract_youtube_id`, `_find_chrome` (Chrome → Chromium → Edge, or `_CHROME_BIN`), `_launch_tv_browser` (`--kiosk --app` + `--autoplay-policy=no-user-gesture-required` + an isolated `--user-data-dir`), `_kill_tv_browser` (kills only the kiosk, matched by profile dir). A play hot-swaps the video over SSE if the `/tv` page is already open (heartbeat <6 s), else launches the kiosk; the fresh page autoplays from `?v=<id>`. Stop broadcasts `yt_command:close` and hard-kills the kiosk. `stat_broadcaster` + `background_video_loop` are gated while `youtube_active` so they don't clobber the YouTube state or start the idle background video over it. Not VPN-gated (ordinary HTTPS, not P2P).
  - **Docs:** new [docs/YOUTUBE.md](docs/YOUTUBE.md); API/GOTCHAS/ARCHITECTURE/PLAN updated. New Milestone 21.

## [3.4.5] — 2026-05-25
- **Every VLC call now reuses one keep-alive connection — the actual fix for the ~10 s play/resume stall.** 3.4.3/3.4.4 assumed only the `in_play` *reply* was slow, but the real problem was that **every** VLC HTTP call (status polls included) paid a per-call TCP setup against VLC's tiny, effectively single-threaded HTTP server. Three background loops (`stat_broadcaster`, `vlc_progress_tracker`, `background_video_loop`) each opened a *fresh* `httpx.AsyncClient()`/socket every 2–3 s, plus the multi-command play burst — constantly churning connections, so each call took seconds and the poll-based UI flip/seek from 3.4.4 was still gated on slow polls.
  - New shared `_vlc_http()` persistent `httpx.AsyncClient` (`base_url` + client-level BasicAuth, `connect=2 s`/`read=5 s` timeouts, a 4-connection keep-alive pool with 30 s expiry). All four VLC callers — `vlc()`, `vlc_status()`, `vlc_playlist_uri()`, `_play_background_video()` — route through it; the per-call `async with httpx.AsyncClient()` blocks are gone. Created in `lifespan` and closed on shutdown (mirrors the `qbit` client), lazily rebuilt if needed. A warm connection from the polling loops is almost always already pooled when a play starts, so commands and status reads stay sub-second.

## [3.4.4] — 2026-05-25
- **Play/resume no longer sits on "buffering" for ~10 s while VLC is already playing.** The real culprit (3.4.3 only sped up the seek): the UI flip to "playing" *and* the resume seek were both gated on `await vlc("in_play")` **returning** — and VLC's HTTP reply to `in_play` lags several seconds behind actual playback (which starts in <1 s). Worse, the two `timeout=5.0` calls in that path (volume pre-roll + in_play) added up to the "at least 10 s" you saw.
  - `_library_play_launch` and `_vlc_relaunch_playlist` (prev/next) now fire `in_play` **detached** and detect the real start by polling the lighter `status.json` endpoint (`_vlc_wait_until_ready`, now file-aware so it can't latch onto the *previously* playing file). The "playing" flip and the resume seek both happen the instant VLC is genuinely on the new file — typically well under a second — instead of when `in_play` finally replies.
  - The remaining-episodes enqueue still waits for the `in_play` call to land first (correct playlist order), but that wait is now invisible: you're already watching. The detached `in_play` task is cancelled if the play is superseded, so a slow reply can't resurrect a stale file.

## [3.4.3] — 2026-05-25
- **Resume on VLC now jumps to your spot the instant VLC has the file open — not after a fixed 3 s wait.** Starting a library item with a saved position used to fire the resume seek on a blind `asyncio.sleep(3)` after `in_play`, so on a local file (where VLC opens in well under a second) you sat at 0:00 watching the start for ~3 s before it jumped, and the seek could be dropped entirely if VLC happened to still be opening. `_library_play_launch` now polls VLC via the new `_vlc_wait_until_ready()` helper (state is `playing`/`paused` **and** `length` is known — i.e. the demuxer is up and a seek will actually take) and seeks immediately, with one guarded re-issue if VLC ignored the first seek right at open. The same wait now gates the `resume_mode="prompt"` offer so it appears as soon as playback is live. Resume is now as snappy as poking VLC from a phone remote.
- **Fixed `/api/version` reporting a stale `3.3.0`**, which left the badge (3.4.2) newer than the server's value so the client auto-reload never triggered. `UI_VERSION` is back in sync with the badge.

## [3.4.2] — 2026-05-25
- **StreamLink's core path is now the box's top priority — controls/UI/VLC-control stay responsive under heavy prep.** Lowering ffmpeg (3.4.1) wasn't enough: the server process itself sat at normal priority, so on a saturated box it got starved while separate apps (e.g. a VLC remote talking straight to VLC) stayed snappy — the on-screen volume button lagged.
  - **The server raises its own priority at startup** (`_raise_own_priority`, called first thing in `lifespan`, so both the HTTP and HTTPS uvicorn processes do it): `HIGH_PRIORITY_CLASS` on Windows (no admin needed; REALTIME deliberately avoided), a negative `nice` on POSIX (needs root/CAP_SYS_NICE — the installed service; degrades gracefully otherwise). The server is I/O-bound, so high priority just lets its short request bursts preempt the encoder without hogging the box.
  - **All background CPU work stays below it.** Prep ffmpeg already ran BELOW_NORMAL/`nice` (3.4.1); the Smart-Skip **analyzer** subprocesses (ffmpeg decode, fpcalc, ffprobe) now do too (`analyzer._lp` + `_LOWPRIO_KW`) — without this they'd inherit the server's HIGH priority and re-introduce lag during an analysis. Net effect: server ≫ VLC/qBit (normal) ≫ prep/analyzer (below normal).

## [3.4.1] — 2026-05-25
- **Keep the dashboard responsive while stream-prep runs.** Prep is CPU-heavy, so beyond the warning/pause/overnight controls (3.4.0) the work now actively yields to the rest of the app rather than just relying on the user to pause it.
  - **ffmpeg runs at lowered OS priority on every platform.** Windows already used `BELOW_NORMAL_PRIORITY_CLASS`; POSIX hosts now prepend `nice -n 10` (`_ffmpeg_nice_prefix`, fork-safe vs `preexec_fn`, no-ops if `nice` is missing). The bulk encode yields CPU to the web server's event loop, VLC, and qBit, so the site stays usable.
  - **No synchronous bursts on the event loop.** The recursive `.part`/old-bundle removals and the bundle size calc in `_run_offline_job` are offloaded with `asyncio.to_thread`, and `/prep-all` + the overnight `_enqueue_library_prep` now `await asyncio.sleep(0)` between files (the per-file `_maybe_start_prep_job` is synchronous), so kicking off prep on a large pack/library no longer briefly stalls other requests. All heavy prep work was already off-loop (ffmpeg/ffprobe as subprocesses); this closes the remaining small windows.

## [3.4.0] — 2026-05-25
- **Stream-prep is now interruptible and can run itself overnight.** Preparing files for on-device streaming pegs the host CPU, so the dashboard now warns about it, lets anyone pause/resume the work, and can do the whole library automatically while nobody's watching.
  - **Lag warning (`static/index.html`):** the first time you tap **Prep for Streaming** (per-item) or a per-row **Prep** in a session, a modal explains that the server will get laggy and can become unusable at times, and points at the pause control. Acknowledged once per session. The interactive "play on this device" path is unaffected (no warning — you want it now).
  - **Pause / Resume from the non-admin UI:** the top-right prep indicator (`#globalPrepBar`) gained **Pause** and **Resume** buttons. Pause asks how to handle the file being encoded right now — **Finish current file, then stop** (no wasted work) or **Stop now** (terminates the in-flight ffmpeg for instant relief; that file restarts from scratch on resume, since HLS prep can't checkpoint mid-file). New endpoints `POST /api/offline-prep/pause {kill}` and `POST /api/offline-prep/resume`.
  - **Overnight auto-prep (admin → System tab):** a new **Overnight Stream Prep** panel with an enable toggle, start/end times (windows may cross midnight), timezone, and an **on-end** choice — **Pause until next window** or **Continue until finished**. During the window the server queues every un-prepped library file; at the end it either holds (letting the in-flight file finish) or runs to completion. Backed by the `overnight_prep_loop` background task and `GET/POST /api/admin/overnight-prep`; config persists under `library.json → settings.overnight_prep`.
  - **Backend model (`main.py`):** bulk/per-item/overnight prep jobs are tagged `queue:"bulk"` and honor a global pause gate in `_run_offline_job`; a paused bulk job releases the single concurrency slot (so an on-demand play-on-device prep can still run) and is re-spawned on resume. `/api/offline-active`, `/prep-status` and `state_snapshot` now expose paused state for the UI.

## [3.3.0] — 2026-05-25
- **Stream-to-device HLS now ships an adaptive-bitrate ladder + an in-player quality menu.** Each bundle previously carried a single video rendition at source resolution, so a weak LAN/Wi-Fi link had no fallback and there was no way to pick a quality. Bundles now emit **Original + 720p + 480p** (each rung capped at the source height — a ≤480p source still gets one rendition, a 720p source gets Original + 480p), all sharing one audio group so switching quality never re-fetches audio.
  - **Player (`static/index.html`):** the on-device player gains a **Res** dropdown (next to Aud/Sub) populated from hls.js `levels` — `Auto` plus each resolution. `Auto` keeps hls.js adapting to bandwidth; a manual pick sets `hls.currentLevel`. Quality is session-only (not persisted). Safari native HLS auto-adapts among the variants but exposes no reliable manual-level API, so its Res row stays hidden (auto-only).
  - **Backend (`main.py`):** `_build_hls_ffmpeg_args` maps the source video once per ladder rung and sets per-output codec/scale (`scale=-2:<h>`, with `maxrate`/`bufsize` VBV caps for realistic master-playlist BANDWIDTH). The original rung still stream-copies when the source is browser-safe H.264 — even when NVENC is present — and only the scaled down-rungs transcode (NVENC on GPU when available, else libx264). `meta.json` and the `offline-prepare` response gain a `videos[]` array.
  - **`OFFLINE_CACHE_VERSION` → `v7-hls-abr`** so existing single-rendition `v6-hls` bundles rebuild on next prep/play; old dirs surface as orphans in Admin → Offline Cache for purge.
  - **Trade-offs:** down-rungs always transcode (slower prep; much faster with NVENC) and bundle storage grows ~1.6–1.9×.

## [3.2.2] — 2026-05-24
- **Bug fix (Windows): `fragLoadError` on `init_video.mp4 → HTTP 404` — the v3.2.1 fix was incomplete.** The new diagnostic alert pinpointed it: the fmp4 init segment 404s on a Windows host. Pinning `-hls_fmp4_init_filename "init_%v.mp4"` fixed the *name* but not the *location*. ffmpeg derives the init segment's output directory by parsing the **playlist** path; on Windows that path has backslashes, the parse fails, and the init file gets written to the server's working directory instead of the bundle `.part/` dir — so it never makes it into the cache and 404s at playback. (A full path on the init arg isn't an option — ffmpeg prepends the playlist dir to it and the encode dies with "Failed to open segment".)
  - **Fix:** every ffmpeg **output** is now a bare filename (init / segments / playlists / subs) and the subprocess runs with `cwd=<bundle .part dir>`, so they all land in the bundle on every OS regardless of path separators. The `-i` source stays absolute. `_build_hls_ffmpeg_args` no longer takes `out_dir`.
  - **`OFFLINE_CACHE_VERSION` → `v6-hls`** so the (still-broken) `v5-hls` bundles from the previous attempt rebuild on next prep/play.

## [3.2.1] — 2026-05-24
- **Bug fix: HLS playback never started — manifest + subtitle/audio dropdowns loaded, then a fatal `fragLoadError` and a black player.** The bundle's manifest and per-rendition playlists fetched fine (so `MANIFEST_PARSED` fired and the dropdowns populated), but the very first media fetch failed. Root cause: `_build_hls_ffmpeg_args` templated every bundle filename *except* the fmp4 init segment, leaving `-hls_fmp4_init_filename` at ffmpeg's default. ffmpeg's own `%v` expansion for the init segment doesn't always match the `#EXT-X-MAP:URI=` it writes into each variant playlist, so the player requested an init segment that didn't exist on disk → 404 → hls.js retried, gave up, and emitted a fatal `fragLoadError` before a single frame decoded.
  - **Fix:** pin `-hls_fmp4_init_filename "init_%v.mp4"` so the init files (`init_video.mp4`, `init_audio_0.mp4`, …) follow the same `name:`-tag scheme as the segments (`seg_video_…`, `seg_audio_0_…`) and the `EXT-X-MAP` URI matches the file on disk on every ffmpeg ≥ 4.3.
  - **`OFFLINE_CACHE_VERSION` → `v5-hls`** so existing (broken) bundles rebuild on next prep/play; old `v4-hls` dirs surface as orphans in Admin → Offline Cache for purge.
  - **Player diagnostics:** the hls.js `ERROR` handler now logs the failing URL + HTTP status to the console and includes the segment name + code in the on-screen alert, so any residual "won't play" is self-diagnosing instead of a bare `fragLoadError`.

## [3.2.0] — 2026-05-20
- **Fix: HLS stream-to-device never worked when the source had subtitles (i.e. essentially always).** The new `logs/hls.log` revealed the real failure: `ffmpeg ... Could not write header ... No streams to mux were specified ... Conversion failed!`. Reproduced locally on the same ffmpeg 8.1.1 — ffmpeg's HLS muxer **cannot** package multi-track WebVTT into a manifest. A single inline subtitle works; two or more (declared as their own `s:N,sgroup:…` variants, as we did) fail unconditionally, for both `fmp4` and `mpegts` segment types. Since virtually every release MKV has many subtitle tracks, HLS prep aborted on every real file.
  - **`_build_hls_ffmpeg_args` rearchitected:** the HLS bundle is now **video + audio only** (fmp4, unchanged and working). Each text subtitle is emitted as a standalone `sub_<i>.vtt` **sidecar** in the same single ffmpeg pass (`-map 0:s:N -c:s webvtt -f webvtt sub_N.vtt`). `meta.json` subtitle entries now carry `file: "sub_<i>.vtt"` instead of an in-manifest `playlist`.
  - **Player (`static/index.html`):** bundle subtitles are wired as `<track>` children of `<video>` exactly like on-disk sidecar subs, unifying the two code paths. `_lpApplySubIdx` now toggles `track.mode` by dropdown key across both the hls.js and Safari-native paths (the old `hls.subtitleTrack` route is gone). New `lp.subTracks` records every created `<track>` in dropdown order.
  - **`OFFLINE_CACHE_VERSION` → `v4-hls`** so any previously-(failed/partial) bundles rebuild on next prep; old `v3-hls` dirs surface as orphans in Admin → Offline Cache.
- **Stream-to-device marked unavailable on macOS hosts.** ffmpeg/ffprobe run as children of the (non-GUI) server process, and macOS TCC blocks them from reading media in the user's protected `~/Downloads` / `~/Desktop` / `~/Documents` — every prep aborts at the probe step with `Operation not permitted` (VLC/qBit work because they're separate, individually-granted apps). The prep endpoints now return a clear 503 (`offline_prepare`) / error (`_maybe_start_prep_job`, `_run_offline_job`) on Darwin, `state_snapshot` exposes `hls_available`, and the UI hides every Prep / On-Device affordance (and routes the play chooser straight to VLC) via a `no-hls` body class.

## [3.1.0] — 2026-05-20
- **Minor feature: real logging for the backend, focused on the HLS offline-prep pipeline.** `main.py` had no `logging` setup at all — background-task failures (most painfully an HLS conversion that died seconds after starting) were swallowed into a 500-char `job["error"]` tail with nothing written to disk, leaving operators no way to find out *why*.
  - **New logger config (`_init_logging`)** wires up a `streamlink` logger with two rotating file handlers — `logs/streamlink_app.log` (everything) and `logs/hls.log` (HLS only, via the `streamlink.hls` child logger) — at 2 MB × 3 backups each, plus a stderr handler capped at `WARNING` so a bulk `/prep-all` doesn't bury the interactive console while the full INFO trail still lands on disk. Idempotent across reloads; `propagate=False` avoids double-logging through uvicorn's root.
  - **`_run_offline_job` instrumented end to end:** logs job START (src/out), the exact ffmpeg command line (shell-quoted), encoder choice + track counts, and DONE (elapsed + bundle size). Every abort path — ffmpeg missing, ffmpeg too old, no video stream — now logs at ERROR. On a non-zero ffmpeg exit it logs the **return code, elapsed time, full command, and the last 300 lines of ffmpeg stderr** to `logs/hls.log` (the UI keeps its short tail). The outer handler uses `hls_log.exception` so crashes carry a traceback.
  - **Latent deadlock fixed as part of the stderr capture:** ffmpeg's stderr is now drained concurrently into a bounded buffer instead of being read only after `proc.wait()`. A chatty ffmpeg could previously fill the OS pipe buffer and block on write while we waited on it — `proc.wait()` would hang forever. Draining as output arrives both prevents that and gives us the complete error to log.

## [3.0.0] — 2026-05-18
- **Major feature: multi-track Stream-to-Device via HLS.** The local-device player now exposes every audio track and every text subtitle from the source MKV, with runtime switching that works across every modern browser. End-to-end rewrite of the offline-prep pipeline.
  - **Backend:** `_run_offline_job` rewritten to emit per-source HLS bundles (`<sha>/master.m3u8` + per-rendition `.m3u8` + fmp4 segments + `meta.json`) instead of single MP4s. One ffmpeg invocation with `-var_stream_map` maps the video, every audio track (each transcoded to AAC stereo at 160 kbps), and every text subtitle (transcoded to WebVTT). Image-based subs (PGS / VOBSUB / DVB) are filtered out by `_ffprobe_full` and reported in `meta.json:skipped_image_subs`. Video stream-copies when the source is already H.264 yuv420p with a non-Hi10P profile; otherwise transcodes via libx264 (or h264_nvenc when available). ffmpeg ≥ 4.3 is now required and enforced via `_ffmpeg_version()` cache — older builds fail-fast with a clear error before launching the encode.
  - **Cache layout:** `OFFLINE_CACHE_VERSION` bumped to `v3-hls`. Cache key is the same `sha256(VERSION|path|mtime|size)[:24]` but now keys a directory (`<sha>/`) instead of a file (`<sha>.mp4`). The old single-MP4 caches map to different keys under v3 and surface as `kind: "legacy"` orphans in Admin → Offline Cache for one-click purge. Per-source storage: bundle ≈ source size + AAC stereo per audio track (a 4 GB BD remux with 3 audios ≈ 4.7 GB vs. 4 GB before).
  - **Endpoints:** `/api/library/offline-cache/{name}` replaced with `/api/library/offline-cache/{cache_key}/{filename}` — strict regex validation on both segments (`[a-f0-9]{24}` + `[A-Za-z0-9._-]+`) prevents path traversal. `/offline-prepare` now returns `master_url` + `audios[]` + `subtitles[]` + `saved_tracks{}` instead of `video_url`. New `POST /api/library/{id}/local-tracks` persists the in-browser player's audio/subtitle picks per-profile per-file as `local_audio_idx` / `local_subtitle_idx` — kept separate from VLC's `audio_track` / `subtitle_track` (ES IDs) which use a different addressing scheme. `update_progress` and `mark_watched` preserve all four track-pref keys across writes.
  - **Admin Offline Cache tab** now walks directories: `_dir_size_bytes` recurses into each `<sha>/` for accurate per-item totals; `_delete_cache_artifacts` removes the bundle dir + any `.part/` staging dir + any legacy MP4 in one operation; the orphan list now includes a `legacy` kind for pre-v3 single-MP4 caches.
  - **Frontend:** ships `static/vendor/hls.min.js` (hls.js v1.5.17, ~100 KB gzipped) bundled locally — lazy-loaded on first local-player open via `_ensureHlsLib()` so the dashboard's initial paint isn't delayed. `_lpLoadIndex` rewritten to branch on `Hls.isSupported()`: hls.js for Chrome / Firefox / Edge / Android Chrome (MSE-based), `<video>.src = master_url` for Safari (native HLS). Audio/sub dropdowns populated from the bundle's `meta.json`; sidecar `.srt`/`.vtt` files coexist with bundle subs as additional dropdown options (using a `"sidecar:N"` index sentinel). `lpSetAudio(idx)` and `lpSetSubtitle(idx)` switch tracks at runtime via `hls.audioTrack` / `hls.subtitleTrack` on hls.js or `AudioTrackList.enabled` / `TextTrackList.mode` on Safari. Each switch POSTs to `/local-tracks` so the pick comes back on next play. Local-player markup gains an `lpAudioRow` next to `lpSubRow`; the row container is hidden when there's exactly one audio track and zero subtitles.
  - **UX trade-offs documented in [docs/GOTCHAS.md](docs/GOTCHAS.md):** segment-boundary seek granularity (~6 s, mostly invisible), ASS/SSA styling loss in WebVTT conversion (libass.js render-on-canvas is the deferred fix in PLAN.md 16.10), image-based subs skipped entirely (PGS / VOBSUB / DVB — use VLC for those), audio always downmixed to AAC stereo (5.1 preservation needs the VLC path), local-player and VLC track prefs intentionally independent.
