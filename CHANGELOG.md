# Changelog

## [2.11.0] — 2026-05-24
- **Minor feature: re-running setup pre-fills your current `.env` values (and doubles as a config health check).** Previously the "Existing .env detected" prompt was all-or-nothing — reuse everything untouched, or re-prompt from scratch with the factory defaults, forcing you to retype every setting (Jackett URL, qBit/VLC passwords, etc.) just to change one. Now choosing to re-prompt pre-fills each field with the value currently in `.env`, so you can press Enter to keep any setting and only type over the ones you want to change. Walking through the prompts now also surfaces every active value, so it works as a quick way to audit the config before, say, re-running setup to apply the Jackett auto-recovery grant.
  - **`gather_config(existing)`** ([setup.py](setup.py)) takes the parsed existing `.env` and defaults each prompt to the stored value, falling back to the factory default on a fresh install (unchanged first-run behaviour).
  - **Secrets are never echoed.** Stored passwords (`JACKETT_PASSWORD`, `QBIT_PASSWORD`, `VLC_PASSWORD`, `ADMIN_PASSWORD`) show a masked `•••••• (Enter = keep)` hint instead of the plaintext value, so re-running setup can't leak them to the terminal/scrollback; pressing Enter still keeps the stored value, and a currently-blank secret shows `(currently blank)`. `ask()` gained a `show_default` override to support this.

## [2.10.1] — 2026-05-23
- **Bug fix: `remote.local` stops resolving after a reboot when run as a system service.** Root cause: the installed service registered mDNS exactly once at startup, but at boot it launches before Wi-Fi has associated and the interface has a LAN IP. `get_local_ip()` returned `""`, so the registration was silently skipped — `remote.local` then never resolved until the next manual relaunch, even though uvicorn (bound to `0.0.0.0`) became reachable by IP the moment the network came up. That's why it worked on first interactive install (network already up) but not after a reboot.
  - **`start_mdns_resilient()` (in `run.py`).** Registers mDNS from a background daemon thread that polls `get_local_ip()` until a LAN IP appears, registers `remote.local`, then re-registers if the IP later changes (DHCP lease, network switch). Polls every 5 s until registered, then every 30 s to watch for changes. Returns a `.close()` handle for shutdown cleanup.
  - **Used by both launch paths.** The generated service wrapper (`daemon.py`) and interactive `run.py` both now call `start_mdns_resilient()` instead of the one-shot `start_mdns()`, keeping them behaviourally identical and fixing the same latent issue when `run.py` is launched before Wi-Fi.
  - **Re-run `python3 run.py --install` to regenerate the service wrapper** with the fix.

## [2.10.0] — 2026-05-23
- **Minor feature: machine reboot + scheduled daily restart (Jackett hard-reset).** A reboot is the only reliable cure for some wedged-Jackett states, so the admin **System** tab now exposes both a manual reboot and an automated nightly one.
  - **Reboot Machine.** New `POST /api/admin/reboot` restarts the whole host (not just the web server). `_reboot_machine()` tries platform-appropriate commands in order — macOS: System Events restart (no sudo from a launchd user agent) → `sudo -n shutdown -r now` → `shutdown -r now`; Linux: `systemctl reboot` → `sudo -n shutdown` → `shutdown`; Windows: `shutdown /r /t 0`. Fires ~0.5 s after the response flushes so the UI gets confirmation first.
  - **Scheduled Restart.** New `scheduled_reboot_loop` background task + `GET`/`POST /api/admin/scheduled-reboot`. Config (`enabled`, `time` HH:MM, `timezone`, `idle_minutes`) persists under `library.json → settings.scheduled_reboot`. At the configured local time (default 00:00 America/Los_Angeles) it checks whether the box was used in the last `idle_minutes`; if idle it reboots, otherwise it waits `idle_minutes` and re-checks, repeating until idle. A persisted `last_fired` date guards against the just-rebooted machine immediately re-arming and looping.
  - **Idle detection.** New `track_activity` middleware stamps `state.last_activity` on mutating requests + search (routine GET polling is ignored). `_machine_in_use()` also treats live VLC playback/pause of non-background content and any active stream/download as "in use", so a reboot never interrupts active watching or a running download.
  - **Auto-recovery requirement.** For the host to come back after a reboot, the user must configure OS auto-login and install the StreamLink system service (`run.py --install`). Documented in the README and the System tab.

## [2.9.0] — 2026-05-20
- **Jackett reliability: detect and recover a hung indexer instead of needing a reboot.** Root cause of "Jackett stops working after a while, only a reboot fixes it": the watchdog's liveness check was a bare TCP port-open, and a hung Jackett keeps its listener socket bound while it stops serving HTTP — so the watchdog thought it was healthy and never restarted it. Even when it did try, `sc.exe start Jackett` is a no-op (1056 ALREADY_RUNNING) on a wedged service, so only a reboot cleared it.
  - **HTTP health check.** The watchdog, `run.py` startup, and a new in-app monitor now probe `GET {INDEXER_URL}/UI/Login` (any HTTP status proves the web stack is alive) via `_http_ok()` instead of a port check. `ServiceSpec` grew a `health_check` hook; `is_alive()` uses it for Jackett (VLC/qBit unchanged), and `start()` waits on it so a restart doesn't tight-loop before Jackett finishes warming up.
  - **Force-down before restart.** A new `pre_restart` hook force-stops the wedged Jackett (`sc stop` + wait for STOPPED, hard-kill fallback; `kill_by_name` off-Windows) before relaunching, so the port frees and the restart actually takes. When no `Jackett` service exists, the watchdog now launches the tray/console exe as a user process instead of giving up.
  - **Permission grant so a non-elevated watchdog can recover the service.** A LocalSystem `Jackett` service can only be `sc` stopped/started by an admin — the reason a non-elevated StreamLink hit a UAC/access-denied wall. `setup.py`'s new `grant_jackett_service_control()` additively grants Authenticated Users `SERVICE_START`+`SERVICE_STOP` via `sc sdset` and sets `sc failure` auto-restart actions (idempotent, best-effort, one UAC prompt; prints the manual commands if elevation is declined). Re-run `setup.py` to apply. Access-denied paths now log an actionable hint.
  - **In-app catch-safe + visibility.** New `jackett_health_monitor` background task polls every 20 s, publishes `state.jackett_ok` (in `state_snapshot()`) and a `jackett_status` SSE event, and — only after ~2 min of sustained local outage (i.e. when no watchdog has recovered it) — calls the reusable `watchdog.restart_jackett()` as a backstop, so it never duels a healthy watchdog.

## [2.8.0] — 2026-05-19
- **Minor feature: bidirectional Handoff (TV ⇄ device), time-synced.**
  - **TV → device.** While a library item is playing on the TV (VLC), a new emerald **Device** button (next to Stop in the player footer, and a **To Device** tile next to Stop in the fullscreen controls) hands playback to the browser that tapped it. Both require a **0.5 s hold** to activate (same `_holdStart` / `.hold-btn` progress-fill pattern as Stop), so an accidental tap can't yank playback off the TV. It captures VLC's live position via a fresh `GET /api/vlc/tracks` read, starts the local `<video>` player on the same item — using the remaining-playlist tail so auto-advance keeps going — seeked to that exact time, and stops VLC. `/api/stop` returns 202, so the device's prep/transcode runs in parallel with the TV tearing down; the resume seek is frozen at the captured time, so the handoff lands on the same frame regardless of how long prep takes. The button shows only during library playback (`is_library_playback && library_item_id`) and is gated by `withInflight("handoff")` against double-taps. To support it, `state_snapshot()` now also publishes `library_item_id` and the full ordered `library_playlist`, and the frontend `app` mirrors both.
  - **Prep gating.** The handoff-to-device button is greyed out with a "Not prepped for on-device streaming" note (footer: tooltip; fullscreen tile: visible sub-label) when the currently-playing file has no `.offline_cache` MP4 and isn't Safari-native — handing off then would stop the TV and drop into a long transcode. Readiness is resolved from `prepFileState` (instant when known) and a per-file `GET /api/library/{id}/prep-status` check (`_handoffReadyState` / `_maybeRefreshHandoffReady`), and the button flips to active automatically once the file is prepped (via the episode picker's Prep / a card's Prep for Streaming). Tapping it while not prepped shows a toast instead of acting.
  - **Device → TV.** The on-device local player now has the mirror-image **To TV** button in its fullscreen header (next to Stop): `lpHandoffToVlc` captures the `<video>` position + remaining playlist tail, tears down the device player (flushing progress), and starts VLC playback at the exact same time via `playLibraryFiles` (`POST /api/library/{id}/play` with `seek_first_to`). Gated by `withInflight("handoff_vlc")`.

## [2.7.0] — 2026-05-19
- **Minor feature: the auto-skip countdown now runs *before* the skip point, and the intro/credits is skipped in full.** Previously the countdown started *at* the intro/credits start and skipped N seconds later, so the first 5 s of an intro (or 10 s of credits) played first. Now the countdown leads *into* the skip point — e.g. for an intro starting at 1:30 the 5-second countdown runs 1:25→1:30 and the skip fires exactly at 1:30, so none of the intro plays. Credits likewise count down over the 10 s before `credits_start` and advance to the next episode the moment credits begin. The countdown is now **position-driven** (`_run_skip_countdown` shows `ceil(target − pos)` and fires at `pos ≥ target`), so it tracks real playback: it freezes while paused and follows seeks, polling every 0.5 s so the popup clears within ~0.5 s when you scrub out of the window. The auto trigger window widened to `[skip_point − lead, …)`; the manual (auto-skip-off) Skip button is unchanged.

## [2.6.2] — 2026-05-19
- **Bug fix:** The countdown popup ("Skipping intro in 1") stayed frozen on the TV after the skip fired — and after any seek that ended the countdown. Root cause: VLC's `marq` filter reads `--marq-file` with `getdelim()`, which returns EOF on an *empty* file, so the filter **keeps the previously-rendered text** instead of blanking it (and logs a read error every refresh tick). Every "clear" wrote an empty string, which was a no-op on screen. `_marquee_write` now clears by writing a single space — a valid non-empty line that forces marq to update but renders no glyph (we draw no background box). `run.py` / `watchdog.py` likewise seed the file with a space, not "". This is what actually made 2.6.1's two-sided-window abort visible on screen.

## [2.6.1] — 2026-05-19
- **Bug fix:** The "Skipping intro in N" countdown popup no longer lingers when the viewer scrubs the progress bar past the intro. `_run_skip_countdown` previously only aborted when the position moved *below* the window floor (a backward seek); a forward seek out of the intro left the popup up and would have yanked playback back to the intro end when the count hit zero. The countdown now takes a `[floor, ceil)` window (intro `ceil = intro end`; credits unbounded) and aborts on leaving it in **either** direction, clearing the popup within ~1 s of the seek.

## [2.6.0] — 2026-05-19
- **Minor feature: on-TV auto-skip countdown popup.** When a profile has auto-skip enabled, Smart Skip no longer cuts instantly — it first shows a countdown directly on the TV (VLC) before acting: **10 s** before auto-advancing past credits, **5 s** before seeking past an intro. The popup renders bottom-right with padding, as opaque white text. Implemented via a VLC `marq` sub-source: VLC is launched (in `run.py`, `watchdog.py`, and `main.py`'s `_restart_vlc_process`) with `--sub-source=marq --marq-file=<repo>/.vlc_marquee.txt --marq-position=10 …`, and `main.py` writes "Skipping intro/credits in N" into that file each second (emptying it clears the popup). Text-only — no global freetype background box — so regular subtitles render unchanged. A dedicated countdown coroutine (`_run_skip_countdown`) polls VLC each tick so it holds while paused and aborts if the viewer seeks out of the window or switches files; Stop / Next / Prev / a new Play all cancel it and wipe the popup. New `state.skip_countdown` field is included in the SSE state snapshot.

## [2.5.0] — 2026-05-18
- **Minor feature: automatic UI cache-bust on version mismatch.** Added `GET /api/version` (returns `{"version": "…"}` with `no-cache` headers) and a `UI_VERSION` constant in `main.py` that must be kept in sync with the version badge in `index.html`. On every page load, a self-invoking script block reads the version baked into the loaded HTML (via the `[data-ui-version]` badge) and compares it against the server response. If the cached page is older than the server's version, the page reloads with a `?_v=<version>` cache-busting query parameter so the browser fetches the updated HTML. A `sessionStorage` guard prevents infinite reload loops.

## [2.4.4] — 2026-05-18
- **Bug fix:** New clients no longer see "No active stream" when they open the remote while VLC is in the middle of real playback — the dashboard now reconciles `state` with VLC on startup. Trigger was a `uvicorn main:app` restart (admin Shut Down, watchdog kick, manual relaunch) while VLC kept running: every `AppState` field reset to its dataclass default (`stream_status="idle"`, `active_title=None`), and `background_video_loop` saw VLC already `state=playing` so it stayed out of the way — leaving every SSE client stuck on "No active stream" until someone pressed Stop and started over. New `_sync_state_from_vlc()` runs in `lifespan` after the volume init: queries `status.json` + `playlist.json`, matches the currently-playing URI against the background-video path (→ sets `background_playing=True` and returns) or every library item's files (→ seeds `active_title`, `library_item_id`, `library_playlist`, `library_current_file`, `active_hash`), and falls back to using the file stem as the title for un-matched playback (external VLC plays, stream-now items whose torrent has been GC'd from `library.json`). `library_profile_id` is intentionally left unset — the original profile isn't recoverable, so `vlc_progress_tracker` skips progress saves and skip offers for the restored session, but title display and next/prev/stop work normally.

This prevents an audio blast when playback starts on a fresh server start — the pre-existing `vlc("in_play")` pre-roll already sends the volume before any media loads, so the correct starting level now flows through cleanly.

## [2.4.0] — 2026-05-18
- **Minor feature: Shut Down Server from the admin panel.** New **System** tab in `/admin` with a single red "Shut Down" button. Posts to a new `POST /api/admin/shutdown` endpoint, which enumerates every `uvicorn main:app` process via psutil and sends SIGTERM — siblings first (so the admin process can finish issuing signals), self last, with a 3 s `os._exit(0)` fallback if uvicorn ignores the signal. Once the HTTP uvicorn exits, `run.py`'s existing `finally` block tears down the HTTPS sibling and the mDNS responder. qBittorrent / Jackett / VLC are not touched — they aren't children of the FastAPI process. Useful for stopping the server cleanly from a phone without SSH-ing into the host.

## [2.3.0] — 2026-05-18
- **Minor feature: hold-to-activate on destructive / navigation controls.** The footer Stop button, the footer Prev / Next episode buttons, and the fullscreen-overlay Stop / Prev / Next tiles now require a 0.5 s press-and-hold to fire instead of a single tap. A short tap is a no-op. While held, a CSS fill grows left-to-right across the button so the user can see the timer counting, then briefly flashes once the action fires. Backed by a generic `_holdStart` / `_holdCancel` helper that wires `pointerdown` / `pointerup` / `pointerleave` / `pointercancel` — works on touch and mouse. ## [2.2.1] — 2026-05-18
- **Bug fix:** Player UI no longer gets stuck on "Loading…" / buffering after a library play, even though VLC is already playing. The 2.2.0 refactor of `/api/library/{id}/play` flipped `state.stream_status` to `"playing"` only *after* the full `in_play` + sequential `in_enqueue` loop completed — so a "continue watching" play that enqueued 50+ remaining episodes left the UI in buffering for many seconds even though VLC had been playing the first episode from the start. The state flip + broadcast now run the instant VLC accepts the first track (the only roundtrip that gates actual playback), and the remaining `in_enqueue` calls run in parallel via `asyncio.gather` so wall time is one roundtrip max regardless of playlist length. Same fix applied to `/api/vlc/prev` and `/api/vlc/next`'s `_vlc_relaunch_playlist` helper.

## [2.2.0] — 2026-05-18
- **Minor feature: slow-network playback responsiveness.** On a slow or flaky link the app used to feel dead for several seconds between "user clicks Play" and "VLC actually opens the file" — the Play response was synchronous on the VLC HTTP roundtrips, the chooser modal closed immediately, and the player UI didn't paint until the SSE `playing` event arrived. Four-part fix:
  1. **Server: non-blocking playback handoff.** `/api/library/{id}/play`, `/api/vlc/prev`, `/api/vlc/next`, `/api/stop`, and `/api/stream` now broadcast a state event (`buffering` / `idle`) and return 202 immediately. The slow VLC `in_play` + `in_enqueue` work (and the qBit deletes on stop/stream) run in background tasks so the response no longer waits on the network at all. A new `state.library_play_task` is tracked and cancelled on subsequent Play / Stop so a slow handoff can't race a newer action.
  2. **Client: optimistic buffering UI.** A new `_optimisticBuffering(label, itemId)` helper flips `app.stream_status` to `buffering`, populates `active_title` from the cached library item, and opens the mobile fullscreen overlay the instant the user clicks Play — no waiting for the server. `renderPlayer` now also auto-opens the overlay on `stream_status === "buffering"` (previously only on `playing`). For library plays the buffering badge says **"Loading…"** and the fullscreen status says **"Starting playback…"** (instead of the misleading "Buffering…" / "Connecting…" copy, which made fully-downloaded episodes look like network problems).
  3. **Client: in-flight guards on Play.** `continueLibraryItem` and `playLibraryFiles` now run under `withInflight("play_${itemId}", …)`, so an impatient double-tap while the server is mid-handoff is dropped client-side instead of racing extra `in_play` requests to VLC. They also bail with a warn-toast if `app._connected === false` (i.e. SSE has been disconnected past the 4 s grace), since the response would otherwise never get a confirming state event.
  4. **Client: visible SSE connection state.** The navbar's SSE pill (`#sseLabel`) now shows **LIVE** / **OFFLINE** on mobile too (previously `hidden sm:block`), and after a 4 s reconnect grace the user gets a "Lost connection to host — reconnecting…" toast plus a guard that blocks new Play actions until SSE re-opens.

## [2.1.6] — 2026-05-17
- **Bug fix:** Idle background video now reliably comes up fullscreen with the taskbar / Dock hidden and every other app minimized when StreamLink is launched as a system service at boot/login. Three-part fix:
  1. VLC is now spawned with `--fullscreen` in all three launch paths (`run.py`, `watchdog.py`, `main.py:_restart_vlc_process`) so it owns the screen from the very first frame instead of relying on a post-`in_play` HTTP toggle.

## [2.1.5] — 2026-05-17

## [2.1.4] — 2026-05-16
- **Bug fix (Windows):** Dropped `/RL HIGHEST` from the scheduled task created by `run.py --install`. On Windows, ports below 1024 do not require admin to bind (the "privileged ports" rule is Unix-only), so the wrapper doesn't need elevation. HIGHEST was actively harmful for Standard User accounts — it made Task Scheduler try to elevate their token at trigger time, which failed silently and left the task registered but never running. Firewall rules (which DO need admin) are now added during `_windows_install` while we hold the UAC-granted admin token.
- **Setup warning:** `setup.py` now detects per-user Python installs on Windows (Microsoft Store python, `AppData\Local\Programs\Python`, etc.) and warns that the resulting venv will not be usable by any other Windows user account. This was the silent root cause of the service-doesn't-run case: when the scheduled task ran as a different user, the venv launcher couldn't read the base python in the installer's user profile and bailed with "Access is denied" before the wrapper could log anything.
- **Docs:** New GOTCHAS entries for Microsoft Store Python and the `/RL HIGHEST` / `/RU` Windows-task pitfalls so the next person doesn't have to re-debug.

## [2.1.3] — 2026-05-16
- **Bug fix (Windows):** `run.py --install` now registers the scheduled task to run as the *console* user (the one actually sitting at the keyboard), not the Admin account whose credentials were entered at the UAC prompt. Previously, installing from an elevated Admin shell while logged in as a regular user would tie the task to the Admin's logon, so the task never fired at the user's logon — the service appeared to install successfully but never started, with no logs written. Detection uses `WTSGetActiveConsoleSessionId` + `WTSQuerySessionInformationW` (works on all Windows editions), with a PowerShell `Win32_ComputerSystem.UserName` fallback. Install output now prints the detected `RunAs` user so it's clear which account the service is tied to.

(Both `setup.py`'s install offer and `run.py --install` already delegate to `daemon.install()`, so the registered task itself is identical between them.)

## [2.0.0] — baseline
- Initial versioned release (previously tracked as alpha1/alpha2).
