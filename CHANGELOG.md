# Changelog

## [4.10.0] — 2026-05-31
- **Fixed: the host stayed laggy long after idle-prep "paused" (sometimes needing a reboot).** The activity-pause only killed the in-flight HLS ffmpeg — the **whisper (AI-subtitle) transcription kept running**, churning CPU/GPU until it finished on its own (potentially very long on CPU), so the box was barely usable in the morning. Whisper (and its audio-extract ffmpeg) are now **cancellable**: an activity-pause kills the running whisper subprocess immediately, the job re-queues as paused, and the GPU→CPU fallback retry is suppressed when the kill was intentional (so it doesn't relaunch). `stt.generate` gained `on_proc` / `cancel_check`; `_pause_prep(kill=True)` now terminates running bulk STT too.
- **New: a viewer shows up → background work stops instantly.** `_activity_kick` pauses+kills bulk prep/STT the moment a viewer interacts **or merely opens the dashboard** (fired from the activity middleware *and* on SSE connect — a page load is a GET that wouldn't otherwise count), instead of waiting up to a 15 s scheduler tick. Idle prep now also treats an open dashboard as "in use" (`_machine_in_use(for_prep=True)`), so it stays paused while you're on the site and only resumes once you leave + the box goes idle. Only applies when idle-prep governs and outside the intentional overnight window; the scheduled reboot still ignores open tabs. Combined with the whisper kill, performance recovers in seconds, not minutes.
- **New: host performance warning + admin health monitor.** A `system_monitor_loop` samples **CPU / RAM / GPU (nvidia-smi) / network** every 5 s and classifies each `ok` / `degraded` / `overloaded`. The dashboard shows a non-blocking **"host is busy — performance may be reduced"** banner whenever the host is degraded/overloaded (auto-clears when it returns to normal), naming what's hot. **Admin → System → System Health** shows the live per-resource status (and whether background prep is running). Exposed via `sys_status` in every `state` SSE event + `GET /api/admin/system-resources`.
- **Docs:** [docs/STT.md](docs/STT.md), [docs/BACKEND.md](docs/BACKEND.md), [docs/ADMIN.md](docs/ADMIN.md), [docs/API.md](docs/API.md), [docs/FRONTEND.md](docs/FRONTEND.md), [docs/GOTCHAS.md](docs/GOTCHAS.md), [PLAN.md](PLAN.md).

## [4.5.0] — 2026-05-28
- **Improved: AI subtitle timing, especially around long pauses.** Generated subs no longer linger across silence or start early. whisper now runs with **DTW token-level timestamp alignment** (`-dtw`) — it warps the decoder's cross-attention against the audio for accurate word boundaries that respect pauses — plus **word-boundary cue splitting** (`-ml 80` + `-sow`), so each cue carries its own accurate timing and a pause becomes a real gap between cues instead of one stretched block. Needs no extra download (alignment heads are built into whisper.cpp) and applies to every model size and both CPU/GPU builds. Regenerate an existing AI track to pick up the better timing.
- **Backend:** `stt._dtw_preset()` maps the configured model (`model_name()`) to its `-dtw` architecture preset via `_DTW_PRESETS`, disabling DTW for any model it can't map confidently (a mismatched preset would error the run); `STT_MAX_LEN` controls cue length. The CPU-fallback retry keeps the new flags.
- **Docs:** [docs/STT.md](docs/STT.md), [docs/GOTCHAS.md](docs/GOTCHAS.md).

## [4.4.0] — 2026-05-28
- **New: idle-triggered auto stream-prep.** In addition to the fixed nightly window, the library can now auto-prep for on-device streaming any time the host has been idle for a configurable number of minutes — no user interaction, VLC playback, active stream, or running download. The moment activity returns it pauses and **discards the in-flight encode** (HLS prep can't checkpoint mid-file, so that file simply restarts from scratch on the next idle stretch). ## [4.0.1] — 2026-05-28
- **Fix: admin → Offline Cache tab froze/crashed the server (stuck on "Loading cache inventory…").** `_build_offline_cache_inventory` did its heavy filesystem walk — `_dir_size_bytes` (recursive `rglob` + `stat` over every segment in every HLS bundle) plus a `stat()` per library file — **synchronously on the asyncio event loop**. With the ABR ladder tripling the segment count per bundle (and a library spread across drives), that walk blocked the loop long enough to wedge the entire server (SSE, VLC polling, every request stalled), which presented as a crash. The request never returned, so the tab sat on "Loading…" forever.
- **Fix:** the inventory now fetches the library + snapshots the job list on the loop, then runs the blocking walk in a worker thread via `asyncio.to_thread` (`_offline_cache_inventory_sync`) — the same discipline `_run_offline_job` already uses for `_dir_size_bytes`/`rmtree`. The job snapshot avoids iterating the live `_offline_jobs` dict off-thread.
- **Docs:** [docs/GOTCHAS.md](docs/GOTCHAS.md) notes never to run a full-cache FS walk inline on the event loop.

## [3.12.2] — 2026-05-27
- **Fix (real one): host *still* crashed on Host Volume changes — pycaw/COM now runs out-of-process.** The 3.12.1 single-thread/never-uninit approach did **not** stop the crash: comtypes still took the whole server down with a native access violation (no traceback) after a few rapid OS-volume calls, even pinned to one COM-initialized thread. A native crash can only be contained by an OS process boundary, so all Windows OS-mixer ops now run in a new long-lived child process, **`winvol_helper.py`**. `main.py` spawns one instance, talks to it over stdin/stdout (one JSON line per request: `{"op":"get"}` / `{"op":"set","pct":N}`), and respawns it transparently if it dies. If COM ever blows up again it kills only the helper — the StreamLink server stays up.
- **Backend:** removed the in-process `_windows_volume_op` + single-thread executor; `set_system_volume`/`get_system_volume` now route through `_winvol_request` on Windows (serialized by an `asyncio.Lock`, 5 s timeout, kill-and-respawn on IO error/timeout). POSIX paths (`osascript`/`pactl`/`amixer`) are unchanged. `_PYCAW_LAST_ERROR` is now populated from the helper's error reply so the YouTube-volume endpoint still surfaces a useful message when pycaw is missing.
- **Docs:** [docs/GOTCHAS.md](docs/GOTCHAS.md) entry rewritten to mandate the subprocess boundary for any future OS-volume caller.

## [3.12.1] — 2026-05-27
- **Fix: host crashed after a few Host Volume changes on Windows (no log output).** Root cause was the Windows volume path doing `CoInitialize → work → CoUninitialize` on **every** call, across arbitrary `asyncio.to_thread` pool workers. Tearing the COM apartment down between calls leaves comtypes' cached interface metadata/proxies dangling; a later access (often GC of a COM pointer on a different pool thread) is a native access violation that kills the whole Python process before Python can raise or log anything — hence "first few work, then the server vanishes with nothing in the logs."
- **Fix:** all Windows volume ops now run on a single dedicated worker thread (`_win_volume_executor`, `max_workers=1`) that calls `CoInitialize` exactly once (tracked via a `threading.local` guard) and **never** `CoUninitialize`s for the life of the process. COM is initialized once, no COM pointer is created on one thread and released on another, and the apartment never tears down. `set_system_volume`/`get_system_volume` route through this executor on Windows; POSIX still uses `asyncio.to_thread` (plain `osascript`/`pactl`/`amixer` subprocesses). This also hardens the existing YouTube-on-TV volume path, which used the same per-call init/uninit.

## [3.11.0] — 2026-05-26
- **Major: HTTPS port (443) is now a reverse proxy to the canonical HTTP app, not a second FastAPI instance.** Previously both ports mounted `main:app` and relied on module-global `AppState` being shared across the two `uvicorn.Server` instances inside one process. In practice this produced intermittent state divergence — clients on `https://remote.local` and `http://<lan-ip>` would see different `playing` state until a refresh — because each ASGI mount has its own startup race, SSE buffer, and event-loop scheduling. Replaced with a strict architecture: port 80 serves `main:app` (the only `AppState` in the process), port 443 serves a tiny new `https_proxy:app` that streams every request to `127.0.0.1:80`. - **`run.py` + `daemon._WRAPPER_CONTENT` updated** to point the port-443 uvicorn at `https_proxy:app` instead of `main:app`. The HTTP server still loads `main:app` and runs the full lifespan (qBit login, background tasks). The `lifespan="off"` workaround on the HTTPS server is gone — the proxy app has its own minimal lifespan (creates/closes the upstream `httpx.AsyncClient`).
- **Cost:** an extra ~ms of localhost latency on HTTPS requests for the proxy hop. Negligible for a home server; well worth the elimination of an entire class of sync bugs.

## [3.4.5] — 2026-05-25
- **Every VLC call now reuses one keep-alive connection — the actual fix for the ~10 s play/resume stall.** 3.4.3/3.4.4 assumed only the `in_play` *reply* was slow, but the real problem was that **every** VLC HTTP call (status polls included) paid a per-call TCP setup against VLC's tiny, effectively single-threaded HTTP server. Three background loops (`stat_broadcaster`, `vlc_progress_tracker`, `background_video_loop`) each opened a *fresh* `httpx.AsyncClient()`/socket every 2–3 s, plus the multi-command play burst — constantly churning connections, so each call took seconds and the poll-based UI flip/seek from 3.4.4 was still gated on slow polls.
  - New shared `_vlc_http()` persistent `httpx.AsyncClient` (`base_url` + client-level BasicAuth, `connect=2 s`/`read=5 s` timeouts, a 4-connection keep-alive pool with 30 s expiry). All four VLC callers — `vlc()`, `vlc_status()`, `vlc_playlist_uri()`, `_play_background_video()` — route through it; the per-call `async with httpx.AsyncClient()` blocks are gone. Created in `lifespan` and closed on shutdown (mirrors the `qbit` client), lazily rebuilt if needed. A warm connection from the polling loops is almost always already pooled when a play starts, so commands and status reads stay sub-second.

## [3.4.2] — 2026-05-25
- **StreamLink's core path is now the box's top priority — controls/UI/VLC-control stay responsive under heavy prep.** Lowering ffmpeg (3.4.1) wasn't enough: the server process itself sat at normal priority, so on a saturated box it got starved while separate apps (e.g. a VLC remote talking straight to VLC) stayed snappy — the on-screen volume button lagged.
  - **The server raises its own priority at startup** (`_raise_own_priority`, called first thing in `lifespan`, so both the HTTP and HTTPS uvicorn processes do it): `HIGH_PRIORITY_CLASS` on Windows (no admin needed; REALTIME deliberately avoided), a negative `nice` on POSIX (needs root/CAP_SYS_NICE — the installed service; degrades gracefully otherwise). The server is I/O-bound, so high priority just lets its short request bursts preempt the encoder without hogging the box.
  - **All background CPU work stays below it.** Prep ffmpeg already ran BELOW_NORMAL/`nice` (3.4.1); the Smart-Skip **analyzer** subprocesses (ffmpeg decode, fpcalc, ffprobe) now do too (`analyzer._lp` + `_LOWPRIO_KW`) — without this they'd inherit the server's HIGH priority and re-introduce lag during an analysis. Net effect: server ≫ VLC/qBit (normal) ≫ prep/analyzer (below normal).
